■ Day2 完全まとめ（HTTPリクエストの中身）


■ ① HTTP通信の構造

通信 = Header + Payload



■ ② Header（付属情報）

■ リクエスト側（ブラウザ → サーバー）

Request URL → どこに送るか
Method → GET（取得） / POST（送信）
Cookie → ログイン情報
User-Agent → どの端末・ブラウザか
Accept → どんな形式で返してほしいか


■ レスポンス側（サーバー → ブラウザ）

Status Code → 成功 / 失敗
Content-Type → 何を返したか（HTML / JSON）
Set-Cookie → Cookieを配布



■ ③ Cookieの仕組み

① ログイン成功
↓
② サーバーがSet-Cookieで発行
↓
③ ブラウザが保存
↓
④ 次回以降、Cookieを自動送信
↓
⑤ サーバーが「ログイン済み」と判断


■ Cookieの中身（例）

SID → ログイン状態のキー（最重要）
SSID → 追加の認証情報
APISID → API通信時の認証材料の一部
NID → 設定・識別（ログイン無関係）



■ ④ POST通信とPayload

POST = データを送る通信
Payload = 送るデータの中身


■ 実際にやったこと

Consoleでfetchを使ってPOST通信を自分で作成
↓
NetworkでPayload確認


■ 本質

ブラウザを使わなくても、自分で通信を作れる



■ ⑤ 通常操作との違い

■ 普通（ボタン）

フロント側の制限あり
・入力チェック
・送信項目固定


■ 手動通信

制限なし
・値を自由に変更
・項目を追加



■ ⑥ 改ざんの基本

・Headerは自由に変更できる
・Payloadも自由に変更できる
・Cookieも送る値を変えられる



■ ⑦ なぜ危険か

サーバーが信用すると成立する


■ 例（権限昇格）

本来：role = user
↓
改ざん：role = admin
↓
DBに保存
↓
ログイン時にadminとして扱われる



■ ⑧ 本質（最重要）

クライアントの情報はすべて信用してはいけない


■ 最終理解

・通信の中身は見える
・通信は自分で作れる
・送るデータは自由に変えられる
・だからサーバー側の防御が必要


■ 一言まとめ

「通信は自由に作れる。だから危険。」
